Overview

  • idea was to create a simple counterpart to Eclipse RCP

  • first ideas for JACP emerged in 2005

  • since 2010 officially an open source project under Apache License 2.0

  • since than the ideas and goals of the project were extended, it became my personal research project ;-)

Why create a counterpart to Eclipse RCP?

Although Eclipse RCP is a great, powerful and established platform the complexity can get a substantial risk

  • major problem: basic concepts are easy to understand but beyond workspace, perspective, view and editor many developers creating "self made" solutions

  • currently Eclipse Foundation is working on e4 project to address some of those complexity issues

What are the goals of JACP?

  1. create a simple API that match following needs:

    • create workbench, perspectives and components
    • communication between components
    • can be implemented in different Toolkits like Swing, SWT, Vaadin or JavaFX 2.0

  2. create a „Actor Like“ approach to run/handle components

    • every component is handled on message call
    • components are handled in separate threads

  3. create Peer-To-Peer based approach to handle component messaging on other/remote instances

    • make components executable outside source JVM
    • create loadbalancing / scheduling to decide where to run component

What is JACP?

  • basically it is a set of interfaces and implementations in Swing, Echo2 and JavaFX2



How does JACP look like?

  • typical... with workbench, perspective and components; But! with a message bus



What are the challenges?

  • Unit testing of asynchronous UI components

  • Access to main application thread (like Event Dispatch Thread in Swing)
